# LiveKit

LiveKit is an open-source framework for building voice and video agents. Agents run on the LiveKit server, which provides a low-latency architecture for real-time communication.

---

## Advantages of LiveKit

- **High-level abstractions** â€” built-in support for speech processing, turn-taking, multimodal events, and LLM integration.

---

## How Does LiveKit Work?

LiveKit consists of two primary components:

- **LiveKit Server** â€” an open-source WebRTC SFU (Selective Forwarding Unit) that orchestrates real-time communication between end users and agents.
- **LiveKit Agents Framework** â€” provides high-level tools for building AI agents, including speech processing, turn-taking, multimodal events, and LLM integration. Agents join rooms as participants and can process incoming media, synthesize output, and interact with users through the same infrastructure that powers all LiveKit applications.

---

## LiveKit Overview

LiveKit's architecture is built around **rooms**, **participants**, and **tracks** â€” virtual spaces where users and agents connect and share media and data across web, mobile, and embedded platforms.

---

## Connecting to LiveKit

You can connect to LiveKit using a `Room` object. A room represents an active LiveKit session. Your app joins a room as a participant; if the room doesn't exist yet, it is created automatically.

Participants can be users, AI agents, devices, or other programs.

- A room is created automatically when the **first participant joins**.
- A room is automatically closed when the **last non-agent participant leaves**.

### Required Parameters

Connecting to a room requires two parameters:

| Parameter | Description |
|-----------|-------------|
| `wsUrl` | The WebSocket URL of your LiveKit server |
| `token` | A unique access token that each participant must use to connect |

### Room Object Attributes

After successfully connecting, the room object exposes two attributes:

| Attribute | Description |
|-----------|-------------|
| `localParticipant` | An object representing the current user |
| `remoteParticipants` | A map of other participants in the room, keyed by their identity |

LiveKit also emits a number of events on the Room object, such as when new participants join or tracks are published.

### Automatic Disconnection

Participants may get disconnected from a room due to server-initiated actions, such as:
- The room is deleted via the `DeleteRoom` API.
- The participant is removed via the `RemoveParticipant` API.

---

## Room Management

A room is a virtual space where real-time communication happens. It can be created manually via the server API, or automatically when the first participant joins. Once the last participant leaves, the room closes after a short delay.

### Installation

```python
uv add livekit-api
```

### Initialize RoomServiceClient

```python
from livekit.api import LiveKitAPI

# Reads LIVEKIT_URL, LIVEKIT_API_KEY, and LIVEKIT_API_SECRET from environment variables
async with api.LiveKitAPI() as lkapi:
    # ... use your client with `lkapi.room` ...
```

### Create a Room

```python
from livekit.api import CreateRoomRequest

room = await lkapi.room.create_room(CreateRoomRequest(
    name="myroom",
    empty_timeout=10 * 60,  # Close after 10 minutes if empty
    max_participants=20,
))
```

### List Rooms

```python
from livekit.api import ListRoomsRequest

rooms = await lkapi.room.list_rooms(ListRoomsRequest())
```

### Delete a Room

```python
from livekit.api import DeleteRoomRequest

await lkapi.room.delete_room(DeleteRoomRequest(
    room="myroom",
))
```

## Participant Management

A **participant** is a user or process taking part in a real-time session. Each participant is represented by a unique developer-provided **identity** and a server-generated **SID**.

### Participant Classes

| Class | Description |
|-------|-------------|
| `LocalParticipant` | Created when the current user connects to a room; represents the local user. |
| `RemoteParticipant` | Created for each remote user that joins the room. |

### Special Participant Types

- **Linked participant** â€” In an agent session, an agent can interact with one participant at a time. The *linked participant* is the one the agent is actively "listening" to.
- **Hidden participant** â€” A participant whose `hidden` permission is set to `true` in their access token. Hidden participants are not visible to others in the room.

### Participant Fields

| Field | Type | Description |
|-------|------|-------------|
| `sid` | string | A UID for this participant, generated by LiveKit server. |
| `identity` | string | Unique identity of the participant, as specified when connecting. |
| `name` | string | Optional display name. |
| `state` | `ParticipantInfo.State` | `JOINING`, `JOINED`, `ACTIVE`, or `DISCONNECTED`. |
| `tracks` | `List<TrackInfo>` | Tracks published by the participant. |
| `metadata` | string | User-specified metadata for the participant. |
| `joined_at` | int64 | Timestamp when the participant joined the room. |
| `kind` | `ParticipantInfo.Kind` | Type of participant. |
| `kind_details` | `ParticipantInfo.KindDetail` | Additional details about participant type. Valid values include `CLOUD_AGENT`, `FORWARDED`, `CONNECTOR_WHATSAPP`, and `CONNECTOR_TWILIO`. |
| `attributes` | string | User-specified attributes for the participant. |
| `permission` | `ParticipantPermission` | Permissions granted to the participant. |

### Types of Participants

You can distinguish between end users and server-side processes using the `kind` field:

| Kind | Description |
|------|-------------|
| `STANDARD` | A regular participant, typically an end-user in your application. |
| `AGENT` | An agent spawned with the Agents framework. |
| `SIP` | A telephony user connected via SIP. |
| `CONNECTOR` | A user connected via a connector (e.g., WhatsApp or Twilio). |
| `EGRESS` | A server-side process recording the session using LiveKit Egress. |
| `INGRESS` | A server-side process ingesting media into the session using LiveKit Ingress. |

### Initialize RoomServiceClient for Participant Management

```python
from livekit.api import LiveKitAPI

# Reads LIVEKIT_URL, LIVEKIT_API_KEY, and LIVEKIT_API_SECRET from environment variables
async with api.LiveKitAPI() as lkapi:
    # ... use your client with `lkapi.room` ...
```

Use `RoomServiceClient` to manage participants in a room.

### List Participants

Requires the `roomList` grant.

```python
from livekit.api import ListParticipantsRequest

res = await lkapi.room.list_participants(ListParticipantsRequest(
    room=room_name
))
```

### Get Participant Details

Requires the `roomAdmin` grant.

```python
from livekit.api import RoomParticipantIdentity

res = await lkapi.room.get_participant(RoomParticipantIdentity(
    room=room_name,
    identity=identity,
))
```

### Update Participant

Requires the `roomAdmin` grant.

```python
from livekit.api import UpdateParticipantRequest, ParticipantPermission

# Promote an audience member to a speaker
await lkapi.room.update_participant(UpdateParticipantRequest(
    room=room_name,
    identity=identity,
    permission=ParticipantPermission(
        can_subscribe=True,
        can_publish=True,
        can_publish_data=True,
    ),
))

# Move them back to audience
await lkapi.room.update_participant(UpdateParticipantRequest(
    room=room_name,
    identity=identity,
    permission=ParticipantPermission(
        can_subscribe=True,
        can_publish=False,
        can_publish_data=True,
    ),
))
```

### Move Participant

Move a participant from one room to another. Requires `roomAdmin` grant.

```python
from livekit.api import MoveParticipantRequest

await lkapi.room.move_participant(MoveParticipantRequest(
    room="<CURRENT_ROOM_NAME>",
    identity="<PARTICIPANT_ID>",
    destination_room="<NEW_ROOM_NAME>",
))
```

### Forward Participant

Forwarding allows you to share a participant's tracks with other rooms. Requires `roomAdmin` and `destinationRoom` grants.

```python
from livekit.api import ForwardParticipantRequest

await lkapi.room.forward_participant(ForwardParticipantRequest(
    room="<CURRENT_ROOM_NAME>",
    identity="<PARTICIPANT_ID>",
    destination_room="<NEW_ROOM_NAME>",
))
```

### Remove Participant

Requires `roomAdmin` grant.

```python
from livekit.api import RoomParticipantIdentity

await lkapi.room.remove_participant(RoomParticipantIdentity(
    room=room_name,
    identity=identity,
))
```

### Mute / Unmute Participant

To mute a participant, first retrieve the `TrackSid` using the `GetParticipant` API, then call `MutePublishedTrack` with the track SID.

```python
from livekit.api import MuteRoomTrackRequest

await lkapi.room.mute_published_track(MuteRoomTrackRequest(
    room=room_name,
    identity=identity,
    track_sid="track_sid",
    muted=True,  # Set to False to unmute
))
```

## Track Management

A **Track** represents a stream of information â€” audio, video, or custom data.

| Concept | Description |
|---------|-------------|
| `Track` | A wrapper around the native WebRTC `MediaStreamTrack`, representing a playable track. |
| `TrackPublication` | If the track is subscribed to by the local participant and available for local playback, it has a `.track` attribute representing the associated `Track` object. |

### TrackPublication Fields

| Field | Type | Description |
|-------|------|-------------|
| `SID` | string | A UID for this track, generated by LiveKit server. |
| `kind` | `Track.Kind` | The type of track: audio, video, or arbitrary data. |
| `source` | `Track.Source` | Source of media: `Camera`, `Microphone`, `ScreenShare`, or `ScreenShareAudio`. |
| `name` | string | The name given to this track when initially published. |
| `subscribed` | bool | Whether this track has been subscribed to by the local participant. |
| `track` | `Track` | If subscribed, the associated `Track` object representing a WebRTC track. |
| `muted` | bool | Whether this track is muted by the local participant. While muted, no new bytes are received from the server. |

### Track Subscription

When a participant subscribes to a track (that hasn't been muted by the publisher), they continuously receive its data. If a participant unsubscribes, they stop receiving media for that track but may resubscribe at any time.

## Webhooks and Events

LiveKit provides two mechanisms for monitoring and responding to changes in rooms, tracks, and participants:

- **Webhooks** â€” server-side notifications sent to your backend when room and participant events occur.
- **Events** â€” client-side event system in the SDKs that allows your application to respond to state changes in real time.

### Managing Webhooks

Webhooks enable your backend to receive real-time notifications about room and participant events.

**Enabling webhooks in self-hosted deployments** â€” set the `webhook` section in your config:

```yaml
webhook:
  # The API key used to sign the message.
  # This must match one of the keys LiveKit is configured with.
  api_key: 'api-key-to-sign-with'
  urls:
    - 'https://yourhost'
```

### Webhook Events

In addition to the fields in the **Payload fields** column, all webhook events include the following common fields:

- `id` â€” UUID identifying the event
- `createdAt` â€” UNIX timestamp in seconds
- `event` â€” Event name

| Event Name | Description | Payload Fields |
|------------|-------------|----------------|
| `room_started` | The first participant joins an empty room. | `room` |
| `room_finished` | Room closes â€” either via `room.close()` or the last participant left and the empty timeout expired. | `room` |
| `participant_joined` | Participant joins (media connection established). Fired after the participant's state changes to active. | `room`, `participant` |
| `participant_left` | Participant leaves a room and all cleanup is complete. | `room`, `participant` |
| `participant_connection_aborted` | Participant connection aborts unexpectedly (e.g., signal connection established but media connection fails). | `room`, `participant` |
| `track_published` | Participant publishes a track. Room and participant objects include only SID, name, and identity. | `room`, `participant`, `track` |
| `track_unpublished` | Participant unpublishes a track. Room and participant objects include only SID, name, and identity. | `room`, `participant`, `track` |
| `egress_started` | Recording/streaming (egress) starts. | `egressInfo` |
| `egress_updated` | Egress updates (e.g., file size change). | `egressInfo` |
| `egress_ended` | Egress ends. | `egressInfo` |
| `ingress_started` | Ingress (external stream) starts. | `ingressInfo` |
| `ingress_ended` | Ingress ends. | `ingressInfo` |

## Connection Events

Connecting to a room happens in two phases:

1. **Signal connection** â€” Establishes the initial signaling channel to exchange metadata and control messages.
2. **Media connection** â€” Establishes the connection that allows the exchange of real-time media and data.

> Media cannot be exchanged until **both phases** are successfully established.

### SDK Events

The following events are available across platform SDKs:

| Event | Description | Event Type |
|-------|-------------|------------|
| `ParticipantConnected` | A remote participant joins the room *after* the local participant has joined. | Room |
| `ParticipantActive` | A remote participant's state changes to active (media connection established). Only available in the JavaScript SDK; for other SDKs, monitor participant state directly. | Room |
| `ParticipantDisconnected` | A remote participant leaves the room. | Room |
| `Reconnecting` | The connection to the server has been interrupted and is attempting to reconnect. | Room |
| `Reconnected` | Reconnection succeeded and the session is active again. | Room |
| `Disconnected` | Disconnected from the room because the room closed or there was an unrecoverable failure. | Room |
| `TrackPublished` | A new track is published to the room after the local participant has joined. | Room, Participant |
| `TrackUnpublished` | A remote participant has unpublished a track. | Room, Participant |
| `TrackSubscribed` | The local participant has successfully subscribed to a remote track. | Room, Participant |
| `TrackUnsubscribed` | A previously subscribed track has been unsubscribed. | Room, Participant |
| `TrackMuted` | A track was muted; fires for both local and remote tracks. | Room, Participant |
| `TrackUnmuted` | A track was unmuted; fires for both local and remote tracks. | Room, Participant |
| `LocalTrackPublished` | A local track was published successfully. | Room, Participant |
| `LocalTrackUnpublished` | A local track was unpublished. | Room, Participant |
| `ActiveSpeakersChanged` | The set of active speakers has changed. | Room |
| `IsSpeakingChanged` | The current participant's speaking status has changed. | Participant |
| `ConnectionQualityChanged` | A participant's connection quality changed. Carries the new `ConnectionQuality` value. | Room, Participant |
| `ParticipantAttributesChanged` | A participant's attributes were updated. | Room, Participant |
| `ParticipantMetadataChanged` | A participant's metadata was updated. | Room, Participant |
| `ParticipantNameChanged` | A participant's display name has changed. | Room, Participant |
| `RoomMetadataChanged` | Room metadata has changed. | Room |
| `DataReceived` | Data received from another participant or the server. | Room, Participant |
| `TrackStreamStateChanged` | A subscribed track's stream state changed (e.g., paused due to bandwidth issues); resumes automatically when conditions improve. | Room, Participant |
| `TrackSubscriptionPermissionChanged` | Track-level subscription permission for the current participant has changed. | Room, Participant |
| `ParticipantPermissionsChanged` | The current participant's permissions have changed. | Room, Participant |
